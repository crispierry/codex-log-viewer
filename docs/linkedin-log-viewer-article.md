# Why I Built Codex Log Viewer: Treating User Intent as a First-Class Audit Trail

I built Codex Log Viewer because I needed a clearer way to understand how AI-assisted software work was actually happening across my projects. The immediate need was practical: I wanted to see how often I was using Codex, which projects were getting the most attention, what kinds of requests I was making, which prompts repeated, how token usage was distributed, and where sessions became unusually large or complex. But the deeper need was not just usage analytics. I wanted a way to see the shape of the work itself.

As AI coding tools become part of everyday engineering, the interaction history starts to matter in a new way. The code diff tells us what changed. The commit message tells us what we decided to preserve. The issue or ticket tells us what we thought we were solving at the beginning. But the user messages sent to an AI agent often contain the most direct record of intent in the middle of the work. They show the corrections, the constraints, the approvals, the design preferences, the moments where the goal changed, and the expectations the agent was supposed to follow.

That is why I believe user messages deserve to be treated as a critical part of the audit trail for AI-assisted development. They are not just chat history. They are operational evidence. They help explain why a program behaves the way it does, why a feature was implemented in a certain direction, why a test was added, why a UI was adjusted, or why a deployment decision was made. In an environment where humans and agents collaborate through natural language, the prompt becomes part of the system history.

The first version of Codex Log Viewer started from that belief, but it had to become useful before it could become interesting. The foundation was a local parser for Codex session logs. I wanted the tool to read the files already on my machine, normalize the records, preserve unknown events, and compute trustworthy metrics without sending private development history to a hosted service. Privacy was not a feature to add later. It was the architecture from the start.

The early work focused on correctness. Codex logs are event streams, and event streams evolve. Some records describe sessions, some describe user messages, some describe assistant responses, some describe token usage, and some contain tool activity or event shapes that are not fully documented. A useful viewer could not assume a perfect schema. It needed to be tolerant, fixture-driven, and honest about what it knew. That meant building parser tests, sanitized fixtures, malformed-line handling, token aggregation rules, and clear separation between exact values and inferred views.

Once the parser and analytics engine were stable enough, the project became a native macOS app. That decision mattered. These logs can contain prompts, code, file paths, command output, screenshots, and business context. I did not want a web dashboard that encouraged uploading sensitive material. I wanted a desktop utility that feels local, inspectable, and private by default. The app starts a private local engine, protects it with an app-owned token, and keeps the primary workflows on the user's machine.

The app evolved through real use. At first, the main questions were basic: which projects did I work on, how many messages did I send, how many tokens did I use, and what sessions were active? Then the workflow became more specific. I needed cross-project search. I needed to select a prompt and see the actual Codex interaction that followed it. I needed Browse, Overview, Search, and Audit views. I needed a way to separate user-authored prompts from generated context wrappers. I needed date filters, project filters, role filters, model filters, session filters, export controls, and a way to inspect warnings and unknown events without pretending the data was cleaner than it was.

One of the most important iterations was the Project Focus view. Seeing raw message counts was useful, but it did not answer what the work was about. So the app now classifies submitted user messages into categories such as implementation, planning, bug fixes, testing, documentation, code review, release work, data analysis, and feedback. That classifier had to be improved through real patterns, not theoretical ones. When too many messages fell into an "other" bucket, that was a signal that the tool was not yet reflecting the work accurately. The point was not to make a chart for its own sake. The point was to turn a pile of interactions into a readable picture of how engineering attention was being spent.

The audit worklog became the clearest expression of the broader idea. Codex Log Viewer can generate a reviewed Markdown worklog for a repository that preserves work-directing user messages, summarizes the response, records what changed, lists verification, and keeps privacy review in the loop. This is not meant to replace commits, pull requests, tests, or human judgment. It complements them. It gives future reviewers a better answer to a question that is becoming more important: what did the human ask the agent to do, and how did the agent respond?

That question matters for quality. If a bug appears later, the user-intent trail can show whether the agent misunderstood the request, whether the human changed direction, whether verification was skipped, or whether the final code drifted away from the original goal. It matters for maintainability, because future contributors can understand not only what exists, but why certain decisions were made. It matters for governance, because teams using AI agents need more than output logs; they need a responsible way to inspect the human-agent collaboration that produced the output.

Today, Codex Log Viewer can parse local Codex logs, group sessions by project, count messages and unique prompts, track token usage by type and model, search messages across projects, browse submitted user messages, inspect the related Codex responses and tool activity, filter operational prompt families, export redacted summaries, and generate audit worklogs for repositories. It is still intentionally local-first. It favors privacy, transparency, and fixture-backed correctness over polish for its own sake.

The future direction is to keep making the audit trail more useful without making it invasive. I want better local indexing for large histories, stronger redacted exports, richer raw-event inspection for advanced users, and more accurate classification of work patterns. I also think there is a broader opportunity here beyond one app or one coding agent. As AI-assisted development becomes normal, we will need better tools for understanding the collaboration itself.

For me, the core lesson from building this is simple: the user message is not disposable. It is the closest artifact we have to intent at the moment work is being directed. If we care about software quality, accountability, and long-term maintainability in AI-assisted engineering, we should preserve that intent carefully, review it thoughtfully, and make it usable without forcing people to give up control of their private development history.
